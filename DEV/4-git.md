## 1. Зачем вообще нужен Git?

### 1.1 Проблема версионирования

Представь, ты делаешь лендинг. Сейчас он выглядит отлично. Решаешь добавить новую фичу — и что-то ломается. Хочешь вернуться назад, но... как?

**Без Git:**
```
landing_v1.html
landing_v2.html
landing_v2_final.html
landing_v2_final_FINAL.html
landing_v2_final_FINAL_2.html
```

Знакомо?

**С Git:**
- Ты делаешь изменения
- Сохраняешь снимок (commit)
- Если что-то сломалось — откатываешься на предыдущий снимок
- Все изменения записаны с комментариями и датами

### 1.2 Git — это машина времени для кода

**Аналогия:**

Git = **сохранения в видеоигре**

- Commit = точка сохранения
- Branch = параллельная реальность (сохранился перед боссом, хочешь попробовать разные стратегии)
- Merge = выбрал лучший вариант и продолжил основную игру

### 1.3 Зачем Git нужен при vibe-coding?

**1. AI генерирует много кода**
- Часто что-то работает, потом ломается
- Нужна возможность вернуться к рабочей версии

**2. Эксперименты**
- Хочешь попробовать новый подход
- Создаёшь отдельную ветку (branch)
- Тестируешь
- Если работает — сливаешь в основную версию
- Если нет — просто удаляешь ветку

**3. Публикация на GitHub**
- Хостинг твоего кода
- Можешь поделиться с другими
- Деплой на VPS
- Портфолио для работодателей

### 1.4 Git vs GitHub

**Важно понять разницу:**

| Что | Git | GitHub |
|-----|-----|--------|
| Тип | Программа на твоём компе | Веб-сайт в интернете |
| Задача | Версионирование (локально) | Хранение кода в облаке |
| Установка | Нужно установить | Регистрация на сайте |
| Работа | Командная строка | Браузер + Git команды |

**Аналогия:**
- **Git** = Microsoft Word (программа на компе)
- **GitHub** = Google Docs (облачное хранилище, можно делиться)

## 2. Как работает Git: базовая модель

### 2.1 Три состояния файлов

Git хранит твои файлы в трёх местах:

```
┌─────────────────────────────────────────────────────┐
│  1. WORKING DIRECTORY (Рабочая директория)          │
│     ↓                                               │
│     Твои файлы, которые ты редактируешь             │
│     Пример: index.html, style.css, bot.py           │
└─────────────────────────────────────────────────────┘
                    ↓ git add
┌─────────────────────────────────────────────────────┐
│  2. STAGING AREA (Индекс)                           │
│     ↓                                               │
│     Подготовленные файлы для следующего commit      │
│     Пример: добавил index.html, но НЕ добавил bot.py│
└─────────────────────────────────────────────────────┘
                    ↓ git commit
┌─────────────────────────────────────────────────────┐
│  3. REPOSITORY (Репозиторий)                        │
│     ↓                                               │
│     Сохранённая история изменений                   │
│     Пример: commit #1, #2, #3...                    │
└─────────────────────────────────────────────────────┘
```

**Аналогия с почтой:**

1. **Working Directory** = пишешь письмо дома
2. **Staging Area** = положил письмо в конверт, готовишь к отправке
3. **Repository** = отправил письмо, оно в архиве почты

### 2.2 Commit — снимок проекта

**Commit** = фотография твоего проекта в конкретный момент времени

Каждый commit содержит:
- Все изменения файлов
- Сообщение (что изменил)
- Автор
- Дата и время
- Уникальный ID (хеш)

**Пример истории commits:**

```
commit 3a7f2c1 - "Добавил форму обратной связи" (сегодня, 14:30)
commit 9b4e8d2 - "Исправил баг с кнопкой" (сегодня, 12:15)
commit 5c1a6f3 - "Создал базовую структуру лендинга" (вчера, 18:00)
```

### 2.3 Branch — параллельные версии

**Branch** (ветка) = отдельная линия разработки

```
main (основная ветка)
  │
  │─── commit 1: Базовый лендинг
  │
  │─── commit 2: Добавил секцию "О нас"
  │
  ├─── feature/new-design (новая ветка)
  │       │
  │       └─── commit 3: Новый дизайн кнопок
  │       │
  │       └─── commit 4: Анимации
  │
  │─── commit 5: Исправил опечатки
  │
  └─── merge feature/new-design → новый дизайн влит в main
```

**Зачем нужны ветки:**
- Тестируешь новую фичу, не трогая рабочую версию
- Несколько человек работают параллельно
- Легко откатиться, если что-то не работает

## 4. Основные команды Git

### 4.1 Инициализация проекта

**Сценарий:** У тебя есть папка с проектом, хочешь начать использовать Git.

```bash
cd /путь/к/твоему/проекту
git init
```

**Что произошло:**
- Создана скрытая папка `.git`
- Теперь Git отслеживает изменения в этой папке

### 4.2 Проверка статуса

```bash
git status
```

**Что показывает:**
- Какие файлы изменены
- Какие файлы в Staging Area
- Какие файлы не отслеживаются

**Пример вывода:**

```
On branch main

Changes not staged for commit:
  modified:   index.html

Untracked files:
  bot.py
```

**Перевод:**
- `index.html` изменён, но НЕ добавлен в Staging
- `bot.py` новый файл, Git его не видит

### 4.3 Добавление файлов в Staging

**Добавить один файл:**

```bash
git add index.html
```

**Добавить все изменённые файлы:**

```bash
git add .
```

**⚠️ Осторожно с `git add .`!**
Может добавить файлы, которые не нужно отслеживать (например, секретные ключи, .env файлы).

### 4.4 Создание commit

```bash
git commit -m "Описание изменений"
```

**Правила хорошего commit message:**

✅ **ХОРОШО:**
```bash
git commit -m "Добавил форму обратной связи"
git commit -m "Исправил баг с отправкой email"
git commit -m "Обновил стили кнопок"
```

❌ **ПЛОХО:**
```bash
git commit -m "изменения"
git commit -m "фикс"
git commit -m "asdfasdf"
```

**Почему важно:**
- Через месяц ты не вспомнишь что делал
- Коллеги не поймут что ты изменил
- Сложно найти где появился баг

### 4.5 Просмотр истории

```bash
git log
```

**Покажет:**
```
commit 3a7f2c1d8e9b4f6a5c2d1e0f (HEAD -> main)
Author: Твоё Имя <your.email@example.com>
Date:   Sat Feb 28 14:30:00 2026

    Добавил форму обратной связи

commit 9b4e8d2c7f1a3b5e6d4c2f0e
Author: Твоё Имя <your.email@example.com>
Date:   Sat Feb 28 12:15:00 2026

    Исправил баг с кнопкой
```

**Короткий формат:**

```bash
git log --oneline
```

**Покажет:**
```
3a7f2c1 Добавил форму обратной связи
9b4e8d2 Исправил баг с кнопкой
5c1a6f3 Создал базовую структуру лендинга
```

## 5. Работа с GitHub

### 5.1 Создание репозитория на GitHub

**Шаги:**

1. Открой [github.com](https://github.com)
2. Нажми **New repository** (зелёная кнопка)
3. Заполни:
   - **Repository name:** `landing-page` (например)
   - **Description:** "Лендинг для веб-студии"
   - **Public/Private:** выбери
   - ❌ НЕ ставь галочку "Add README"
4. Нажми **Create repository**

### 5.2 Связка локального проекта с GitHub

**GitHub покажет команды, вот что нужно:**

```bash
# Добавляем удалённый репозиторий
git remote add origin https://github.com/твой-username/landing-page.git

# Отправляем код на GitHub
git push -u origin main
```

**Что произошло:**
- Твой код теперь на GitHub
- Можешь открыть `https://github.com/твой-username/landing-page` и увидеть файлы

### 5.3 Клонирование чужого проекта

```bash
git clone https://github.com/username/project-name.git
```

**Что произошло:**
- Скачал весь проект с GitHub
- Создана папка `project-name`
- Вся история commits тоже скачана

### 5.4 Обновление кода (Pull / Push)

**Push** = отправить свои изменения на GitHub

```bash
git push origin main
```

**Pull** = скачать изменения с GitHub

```bash
git pull origin main
```

**Когда использовать:**
- **Push:** сделал commit локально, хочешь загрузить на GitHub
- **Pull:** работаешь с нескольких компов или в команде, нужны последние изменения

## 6. Работа с ветками (Branches)

### 6.1 Зачем нужны ветки при vibe-coding

**Сценарий:**

Ты делаешь лендинг. AI сгенерировал рабочую версию. Хочешь попробовать новый дизайн, но не сломать текущий.

**Решение:**
1. Создаёшь ветку `feature/new-design`
2. Экспериментируешь в этой ветке
3. Если понравилось → сливаешь в `main`
4. Если нет → удаляешь ветку

### 6.2 Создание и переключение веток

**Создать ветку:**

```bash
git branch feature/new-design
```

**Переключиться на ветку:**

```bash
git checkout feature/new-design
```

**Создать и сразу переключиться (короткий вариант):**

```bash
git checkout -b feature/new-design
```

### 6.3 Просмотр веток

```bash
git branch
```

**Покажет:**
```
  main
* feature/new-design  ← текущая ветка (звёздочка)
```

### 6.4 Слияние веток (Merge)

**Сценарий:** Новый дизайн готов, хочешь влить в `main`.

```bash
# Переключаемся на main
git checkout main

# Вливаем изменения из feature/new-design
git merge feature/new-design
```

**Что произошло:**
- Все изменения из `feature/new-design` теперь в `main`
- История commits сохранена

### 6.5 Удаление ветки

```bash
# Удалить локальную ветку
git branch -d feature/new-design

# Удалить удалённую ветку на GitHub
git push origin --delete feature/new-design
```

## 7. .gitignore — что НЕ нужно коммитить

### 7.1 Зачем нужен .gitignore

**Проблема:**

У тебя есть файлы, которые НЕ должны попасть в Git:
- `.env` (секретные ключи)
- `node_modules/` (библиотеки, весят много)
- `__pycache__/` (кеш Python)
- `.DS_Store` (системные файлы macOS)

### 7.2 Если уже закоммитил .env (частая ошибка!)

**Проблема:** Ты случайно закоммитил `.env` с секретными ключами.

**Решение:**

```bash
# Удалить из Git (но оставить локально)
git rm --cached .env

# Добавить в .gitignore
echo ".env" >> .gitignore

# Закоммитить изменения
git add .gitignore
git commit -m "Удалил .env из отслеживания"
git push
```

**⚠️ ВАЖНО:** Если ключи уже на GitHub, их нужно **сменить**! GitHub сканирует репозитории на утечки.

## 8. Частые кейсы

### 8.1 "Хочу откатить файл к предыдущей версии"

**Проблема:** Сломал `index.html`, хочу вернуть как было.

**Решение:**

```bash
# Откатить к последнему commit
git checkout -- index.html
```

**Или вернуть к конкретному commit:**

```bash
# Найти нужный commit
git log --oneline

# Вернуть файл из commit 5c1a6f3
git checkout 5c1a6f3 -- index.html
```

### 8.2 "Конфликт при merge"

**Проблема:**

```bash
git merge feature/new-design
# CONFLICT (content): Merge conflict in index.html
```

**Что произошло:**
- В `main` и `feature/new-design` изменили одну и ту же строку
- Git не знает какую версию оставить

**Решение:**

1. Открой `index.html`
2. Увидишь:

```html
<<<<<<< HEAD
<button>Старая кнопка</button>
=======
<button>Новая кнопка</button>
>>>>>>> feature/new-design
```

3. Удали маркеры `<<<<<<<`, `=======`, `>>>>>>>` и оставь нужную версию:

```html
<button>Новая кнопка</button>
```

4. Закоммить:

```bash
git add index.html
git commit -m "Разрешил конфликт в index.html"
```

### 8.3 "Хочу полностью откатиться к предыдущему commit"

**Проблема:** Последние 2 commit — мусор, хочу их удалить.

**Решение (локально, НЕ на GitHub):**

```bash
# Откатиться на 2 commit назад (удалив изменения)
git reset --hard HEAD~2
```

**⚠️ ОПАСНО:** Все изменения из последних 2 commits потеряны!

**Если нужно откатить, но сохранить файлы:**

```bash
git reset --soft HEAD~2
```

## 9. Git в работе с AI (практика vibe-coding)

### 9.1 Workflow с AI инструментами

**Типичный процесс:**

1. **Создаёшь ветку для эксперимента**

```bash
git checkout -b feature/ai-generated-form
```

2. **AI генерирует код**

```bash
# Claude Code создал form.html, form.js
git add form.html form.js
git commit -m "AI сгенерировал форму обратной связи"
```

3. **Тестируешь → не работает → правишь промпт**

```bash
# AI исправил баг
git add form.js
git commit -m "Исправил валидацию email"
```

4. **Тестируешь → работает → вливаешь в main**

```bash
git checkout main
git merge feature/ai-generated-form
git push origin main
```

### 9.2 Коммитить после каждого AI-изменения?

**Нет, не обязательно.**

✅ **ХОРОШАЯ ПРАКТИКА:**
- Commit после **завершённой фичи**
- Commit после **исправления бага**
- Commit когда **всё работает**

❌ **ПЛОХАЯ ПРАКТИКА:**
- Commit после каждого промпта
- 50 commits вида "попытка 1", "попытка 2"

**Правило:** Commit = атомарное изменение (одна фича, один баг).

### 9.3 Использование веток для экспериментов

**Зачем:**
- AI иногда генерирует код, который ломает всё
- Можешь безопасно тестировать
- Если не работает → просто удаляешь ветку

**Пример:**

```bash
# Эксперимент с новой библиотекой
git checkout -b experiment/new-library
# AI добавил библиотеку → не работает
git checkout main
git branch -d experiment/new-library  # удалили, как будто не было
```

## 10. Работа в команде (базовые правила)

### 10.1 Pull Request (PR)

**Что это:**
- Ты делаешь изменения в своей ветке
- Отправляешь на GitHub
- Просишь коллег проверить код
- Если ОК → вливаешь в `main`

**Workflow:**

```bash
# 1. Создаёшь ветку
git checkout -b feature/new-button

# 2. Делаешь изменения
git add .
git commit -m "Добавил кнопку 'Связаться'"

# 3. Отправляешь на GitHub
git push origin feature/new-button

# 4. На GitHub создаёшь Pull Request
# 5. Коллеги ревьюят
# 6. Если всё ОК → Merge
```

### 10.2 Правило: НИКОГДА не пушить в main напрямую

**Почему:**
- `main` = production-ready версия
- Должна всегда работать
- Изменения через PR

**Защита:**

На GitHub можно настроить **Branch protection rules**:
- Запретить прямой push в `main`
- Требовать PR
- Требовать проверку коллег

### 10.3 Синхронизация с командой

**Перед началом работы:**

```bash
git pull origin main
```

**Это избежит конфликтов.**

**Пример:**

❌ **ПЛОХО:**
```bash
# Утром не сделал pull
# Работал весь день
# Вечером делаешь push → конфликты на каждом файле
```

✅ **ХОРОШО:**
```bash
# Утром
git pull origin main

# Работаешь
# Вечером
git push origin main
```

## 12. Полезные ресурсы

### 12.1 Шпаргалка команд

```bash
# === БАЗОВЫЕ ===
git init                           # Инициализировать проект
git status                         # Проверить статус
git add <файл>                     # Добавить файл в staging
git add .                          # Добавить все файлы
git commit -m "сообщение"          # Создать commit
git log                            # История commits
git log --oneline                  # Короткая история

# === ВЕТКИ ===
git branch                         # Список веток
git branch <название>              # Создать ветку
git checkout <название>            # Переключиться на ветку
git checkout -b <название>         # Создать и переключиться
git merge <название>               # Влить ветку в текущую
git branch -d <название>           # Удалить ветку

# === УДАЛЁННЫЕ РЕПОЗИТОРИИ ===
git remote add origin <URL>        # Связать с GitHub
git push origin <ветка>            # Отправить на GitHub
git pull origin <ветка>            # Скачать с GitHub
git clone <URL>                    # Клонировать проект

# === ОТКАТ ИЗМЕНЕНИЙ ===
git checkout -- <файл>             # Откатить файл
git reset --soft HEAD~1            # Откатить последний commit (сохранить изменения)
git reset --hard HEAD~1            # Откатить последний commit (удалить изменения)
```

## 13. Частые вопросы (FAQ)

### Q1: Нужно ли запоминать все команды?

**A:** Нет. Запомни основные 5-7 команд. Остальное —ググл или `git --help`.

**Основные команды:**
```bash
git add .
git commit -m "сообщение"
git push origin main
git pull origin main
git status
```

---

### Q2: Что делать если всё сломалось?

**A:** Не паникуй. Git почти никогда не удаляет данные.

**Шаги:**
1. `git status` — посмотри что произошло
2. `git log` — найди последний рабочий commit
3. `git checkout <hash>` — вернись к нему
4. Если совсем плохо → спроси в чате воркшопа или ググл

---

### Q3: Когда делать commit?

**A:** Когда завершил **атомарное изменение**.

✅ **ХОРОШО:**
- Добавил форму → commit
- Исправил баг → commit
- Обновил стили → commit

❌ **ПЛОХО:**
- После каждого промпта AI → не commit
- "Сохранюсь на всякий случай" → не commit

---

### Q4: Что лучше: GUI или командная строка?

**A:** Начни с командной строки. Потом используй GUI.

**Почему:**
- Понимание что происходит под капотом
- 90% туториалов — для командной строки
- GUI могут сломаться, команды — всегда работают

---

### Q5: Можно ли удалить commit из истории?

**A:** Да, но **ТОЛЬКО если НЕ отправил на GitHub**.

```bash
# Удалить последний commit (локально)
git reset --hard HEAD~1
```

**⚠️ НЕ делай это после `git push`!** Сломаешь историю для команды.

---

### Q6: Как вернуть удалённый файл?

**A:**

```bash
# Если удалил, но НЕ закоммитил
git checkout -- <файл>

# Если удалил и закоммитил
git log --oneline  # найди commit где файл был
git checkout <hash> -- <файл>
```

### Q7: Нужно ли коммитить `.env` файлы?

**A:** **НЕТ! НИКОГДА!**

**Почему:**
- В `.env` секретные ключи (API tokens, пароли БД)
- Если закоммитишь → ключи на GitHub → все видят

**Правильно:**

1. Создай `.env.example`:

```
API_KEY=your_api_key_here
DB_PASSWORD=your_password_here
```

2. Добавь `.env` в `.gitignore`:

```
.env
```

3. Коммить только `.env.example`

**A:** Сколько угодно. Git легковесный.

**Типичные ветки:**
- `main` — основная
- `feature/new-form` — новая фича
- `bugfix/email-validation` — исправление бага

### Q9: Как удалить репозиторий с GitHub?

**A:**

1. Зайди на GitHub
2. Открой репозиторий
3. **Settings** → **Danger Zone** → **Delete this repository**
4. Подтверди

**⚠️ Это навсегда!** Убедись что скачал код локально.
