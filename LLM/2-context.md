## 1. КОНТЕКСТНОЕ ОКНО — РАБОЧАЯ ПАМЯТЬ LLM

### Что это такое?

**Контекстное окно** = рабочая память модели = всё, что она "видит" прямо сейчас

```
Аналогия с компьютером:

Контекстное окно = RAM (оперативная память)
─────────────────────────────────────────────
RAM 2GB  → можно открыть 2 вкладки Chrome
RAM 16GB → можно открыть 50 вкладок

Контекст 8K токенов   → короткий чат
Контекст 200K токенов → длинный документ
```

### Визуализация контекстного окна

```
┌─────────────────────────────────────────┐
│  КОНТЕКСТНОЕ ОКНО (200,000 токенов)     │
├─────────────────────────────────────────┤
│ Системный промпт         [2,000 токенов]│
│ Правила проекта          [1,000 токенов]│
│ История чата            [30,000 токенов]│
│ Загруженные файлы       [50,000 токенов]│
│ Определения инструментов[15,000 токенов]│
│ Ваш текущий запрос       [2,000 токенов]│
│ ─────────────────────────────────────── │
│ ИСПОЛЬЗОВАНО:           100,000 токенов  │
│ ОСТАЛОСЬ ДЛЯ ОТВЕТА:    100,000 токенов  │
└─────────────────────────────────────────┘
```

|--------|----------|-------------|----------|
| **GPT-4 Turbo** | 128K токенов | Средняя книга | ~300 |
| **Claude 3.5 Sonnet** | 200K токенов | Большая книга | ~500 |

| **Gemini 2.5 Pro** | 1M токенов | Полное собрание | ~2000 |

**Формула:** 1 страница ≈ 400 токенов (русский текст)

---

### 🔥 Важная концепция: Context Rot (гниение контекста)

**Исследование показало:** Чем больше контекст, тем хуже модель работает!

```
Качество ответа модели:

100%  ████████████           ← Контекст 0-32K
       ↓
 85%  ████████              ← Контекст 32-64K
       ↓
 70%  ██████                ← Контекст 64-128K
       ↓
 55%  ████                  ← Контекст 128K+
```

**Вывод:** Больше контекста ≠ лучше работа!

## 2. ЧТО СЪЕДАЕТ КОНТЕКСТ

### Формула контекстного окна

```
КОНТЕКСТНОЕ ОКНО =
    Системный промпт (System)
  + Правила проекта (CLAUDE.md, AGENTS.md)
  + Определения инструментов (MCP, tools)
  + История чата (все предыдущие сообщения)
  + Загруженные файлы (@mentions)
  + Результаты выполнения (tool results)
  + Ваш текущий запрос
  + Резерв для ответа (output)
```

### 🚨 Что происходит при переполнении?

#### Вариант 1: Автоматическая обрезка (Truncation)

```
До:  [Сообщение 1][Сообщение 2][Сообщение 3]...[Сообщение 50]

После переполнения:
     [Сообщение 20][Сообщение 21]...[Сообщение 50]
     
❌ Проблема: Старый контекст удалён!
Модель "забыла" начало разговора
```

#### Вариант 2: Ошибка

```
Error: Context window exceeded
Maximum context length: 200000 tokens
Your request: 250000 tokens

❌ Запрос отклонён
```

---

### Проблема: "Lost in the Middle" (потерялось в середине)

**Исследование NOLIMA (2025):** Модель лучше помнит начало и конец, хуже — середину

```
Точность ответов модели:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
█████████ НАЧАЛО документа       90%
──────────────────────────────────
         СЕРЕДИНА документа      60%
──────────────────────────────────
█████████ КОНЕЦ документа        85%
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Практический пример:**

```
Вы загрузили 50 файлов проекта:

Файл №1:  index.js        ← ✅ Модель помнит хорошо
Файл №2:  App.tsx         ← ✅ Помнит
Файл №5:  auth.ts         ← ⚠️ Начинает забывать
Файл №25: utils.js        ← ❌ Высокий шанс "потерять"
Файл №45: config.ts       ← ⚠️ Лучше помнит
Файл №50: constants.ts    ← ✅ Помнит хорошо
```

**Вывод:** 
- Важную информацию размещай **в начале** или **в конце**
- Не надейся на середину длинного контекста!

## 3. PROMPT CACHING (ЭКОНОМИЯ ДЕНЕГ)

### Что такое Prompt Caching?

**Кэширование промпта** = сохранение повторяющихся частей для повторного использования

```
Принцип работы:

Запрос №1:
┌────────────────────────────────────────┐
│ [Системный промпт 10,000 токенов]      │ ← Вычисляется
│ [Правила проекта 5,000 токенов]        │ ← Вычисляется
│ [Ваш вопрос 100 токенов]               │ ← Вычисляется
└────────────────────────────────────────┘
Стоимость: $0.50

Запрос №2 (в течение 5 минут):
┌────────────────────────────────────────┐
│ [Системный промпт ✅ CACHED]           │ ← Из кэша!
│ [Правила проекта ✅ CACHED]            │ ← Из кэша!
│ [Новый вопрос 100 токенов]             │ ← Вычисляется
└────────────────────────────────────────┘
Стоимость: $0.01

💰 Экономия: 50x!
```

---

### Реальные цифры

#### Claude API (Anthropic)

| Токены | Без кэша | С кэшем | Экономия |
|--------|----------|---------|----------|
| Input | $3.00 / 1M | $0.30 / 1M | **10x** |
| Cache write | — | $3.75 / 1M | — |
| Cache read | — | $0.30 / 1M | **10x** |

**Пример расчёта:**

```
Сессия из 10 запросов:
Системный промпт: 20,000 токенов (не меняется)
Вопросы: по 500 токенов каждый

БЕЗ КЭША:
(20,000 + 500) × 10 запросов = 205,000 токенов
205,000 × $3.00 / 1M = $0.62

С КЭШЕМ:
Первый запрос: 20,000 (write) + 500 = $0.08
9 следующих: 20,000 (cached) + 500 × 9 = $0.07
Итого: $0.15

💰 Экономия: $0.47 (75%!)
```

## 4. СТРАТЕГИИ УПРАВЛЕНИЯ КОНТЕКСТОМ

### Стратегия 1: Начинай новый чат для новых задач

```
❌ ПЛОХО: Один чат на весь день

09:00 "Помоги с React"
10:00 "Теперь про Python"
11:00 "А теперь про DevOps"
14:00 "Опять React"

Проблема:
- Контекст переполнен
- Модель путается между задачами
- Низкое качество ответов
```

```
✅ ХОРОШО: Новый чат для каждой задачи

Чат 1: React компонент
Чат 2: Python скрипт
Чат 3: Docker настройка
Чат 4: React тесты

Плюсы:
- Чистый контекст
- Модель фокусируется
- Высокое качество
```

---

### Стратегия 2: Суммаризация (Summarization)

**Когда использовать:** После 15-20 сообщений

```
Ситуация:
История чата: 40,000 токенов (20 сообщений)

Промпт суммаризации:
"Суммируй наш разговор в 5 ключевых пунктов"

Результат:
"1. Создали компонент Auth
 2. Добавили валидацию email
 3. Интегрировали с Firebase
 4. Написали тесты
 5. Исправили баг с редиректом"

Экономия: 40,000 токенов → 500 токенов (80x!)
```

**После суммаризации:**
- Начни новый чат
- Вставь суммаризацию в начало
- Продолжай работу

---

### Стратегия 3: Используй @mentions осознанно

```
❌ ПЛОХО: Загрузить весь проект

@src/
  ├─ components/ (50 файлов)     ← 100,000 токенов
  ├─ pages/ (30 файлов)          ← 60,000 токенов
  ├─ utils/ (20 файлов)          ← 40,000 токенов
  └─ ...

Итого: 200,000 токенов = ВЕСЬ контекст!
```

```
✅ ХОРОШО: Только релевантные файлы

Задача: Исправить логин

@components/Auth/Login.tsx       ← 5,000 токенов
@api/auth.ts                     ← 3,000 токенов
@utils/validation.ts             ← 2,000 токенов

Итого: 10,000 токенов = 5% контекста ✅
Остаётся 190,000 для работы!
```

### Стратегия 4: Директорные правила (Cursor, Claude Code)

**Проблема:** Один CLAUDE.md для всего проекта съедает много токенов

**Решение:** Создавай правила для каждой директории

```
Структура проекта:

/project
  CLAUDE.md                    ← Общие правила (1,000 токенов)
  
  /frontend
    CLAUDE.md                  ← Правила для фронта (800 токенов)
    /components
    /pages
  
  /backend
    CLAUDE.md                  ← Правила для бэка (900 токенов)
    /api
    /services
  
  /docs
    CLAUDE.md                  ← Правила для доков (500 токенов)
```

**Как это работает:**

```
Работаешь в /frontend:
✅ Загружается: 
   - /project/CLAUDE.md (1,000)
   - /frontend/CLAUDE.md (800)
   Итого: 1,800 токенов

❌ НЕ загружается:
   - /backend/CLAUDE.md
   - /docs/CLAUDE.md

💡 Экономия: 1,400 токенов!
```
